# MAJU Intelligence v1 플로우 검증 기준

이 문서는 MAJU Intelligence v1을 더 만들기 전에 제품 흐름이 맞는지 검증하기 위한 기준입니다.

## 1. 첫 진입 목적

사용자가 처음 들어왔을 때 이해해야 하는 메시지는 하나입니다.

> 관리자가 회사와 계정을 생성하고, 사용자는 설정에서 회사 기준값을 수정한 뒤 거래처/매출 데이터를 쌓아 현재 상태와 다음 영업 방향을 볼 수 있다.

초기 화면은 CRM 기능 소개가 아니라 사용자가 어떤 데이터를 어떤 순서로 넣어야 하는지 보여줘야 합니다.

앞의 가이드:

1. 관리자가 회사와 고객사 계정을 생성한다.
2. 사용자는 설정에서 회사 기본정보와 물류 출발지를 수정한다.
3. 사용자는 거래처 정보를 기입하고 히스토리를 관리한다.
4. 사용자는 매출 정보를 기입하거나 대량 등록한다.

기대효과:

1. 사용자는 현재 회사 현황을 알 수 있다.
2. 사용자는 현재 거래처의 분포도와 히스토리 관리를 할 수 있다.
3. 거래처 분포와 히스토리는 지도 시각화로 연결한다.

## 2. 데이터 등록 분류

등록은 반드시 두 종류로 나눕니다. 엑셀 업로드는 등록 방법 중 하나이며, 소량 데이터는 화면에서 수기로 입력해 저장할 수 있어야 합니다.

### 0. 회사/계정 생성과 설정

목적: 회사 생성은 사용자가 직접 하는 가입 플로우가 아니라 MAJU 관리자가 먼저 생성합니다.

원칙:

- 관리자가 고객사 회사와 최초 관리자 계정을 생성합니다.
- 사용자는 로그인 후 설정에서 회사명, 담당자, 업태, 물류 출발지 주소를 수정합니다.
- 물류 출발지 주소는 거래처 배송주소까지의 거리/시간/동선 계산 기준값입니다.
- 회사 생성은 관리자 화면의 책임이고, 회사 정보 수정은 고객사 설정 화면의 책임입니다.

### A. 거래처 마스터 등록

목적: 거래처 기본값을 1회 등록하고, 이후 보완/수정합니다.

필수/주요 데이터:

- 거래처/매장 상호명
- 사업자등록번호
- 사업자등록증 정보
- 개업일
- 대표자명
- 사업자 정보
- 통장 사본
- 배송주소
- 연락처
- 이메일
- 생년월일
- 회사 출발지 기준 티맵 거리/시간/경로 캐시
- 사업자 상태 체크 결과

원칙:

- 같은 거래처는 새로 만들지 않고 업데이트합니다.
- 사업자등록번호가 있으면 가장 강한 key로 사용합니다.
- 사업자등록번호가 없으면 상호명 + 주소 기반 key를 사용합니다.
- 티맵 거리/시간은 사람이 입력하는 값이 아니라 API 계산 결과를 저장합니다.

### B. 매출 데이터 등록

목적: 일/월/분기/반기/연 단위 매출 분석과 품목 이탈 분석을 수행합니다.

필수/주요 데이터:

- 거래처/매장 상호명
- 사업자등록번호 또는 거래처 key
- 매출일자
- 매출금액
- 품목명
- 수량
- 지역
- 주소

원칙:

- ERP마다 헤더명이 달라도 컬럼 매핑으로 흡수합니다.
- 한 번 맞춘 ERP별 매핑은 프리셋으로 저장해 재사용합니다.
- 매출 엑셀은 거래처 마스터의 배송거리/주소 기본값을 임의로 덮어쓰지 않습니다.
- 매출 데이터는 거래 내역으로 쌓고, 분석 리포트는 누적 데이터 기준으로 갱신합니다.

## 3. 등록 방법과 ERP별 컬럼 매핑

MVP에서 반드시 보여줘야 하는 흐름:

1. 등록 유형 선택: 거래처 마스터 또는 매출 데이터
2. 등록 방법 선택: 수기 입력 또는 엑셀 대량 등록
3. 수기 입력은 필수값을 바로 저장
4. 엑셀 등록은 자동 매핑 시도
5. 사용자가 ERP별 헤더를 직접 보정
6. 매핑 저장
7. 다음 엑셀 등록 때 같은 ERP 프리셋 자동 적용

필요한 기능:

- ERP/유통사별 매핑 프리셋 이름
- 업로드 유형별 프리셋 분리
- 필수 컬럼 누락 표시
- 수동 매핑 수정
- 매핑 저장 및 재사용

## 4. 저장/업데이트 체감

업로드 후 사용자는 아래를 바로 알아야 합니다.

- 몇 개 row를 읽었는지
- 신규 거래처가 몇 개 추가됐는지
- 기존 거래처가 몇 개 업데이트됐는지
- 중복/누락/오류가 몇 개인지
- 마지막 업데이트 시간이 언제인지
- 어떤 리포트가 새로 생성됐는지

대표 화면에서는 “저장됐다”보다 “이 데이터가 어떤 가치로 바뀌었는가”를 보여줘야 합니다.

예:

- 회사 건강도 변화
- 이번 달 매출 변화
- 이탈 위험 거래처
- 품목 이탈
- 신규 확장 가능 지역
- 오늘 우선 방문 대상

## 5. 대표/중간관리자 대시보드 기준

대시보드의 목적은 기능 탐색이 아니라 현황 판단입니다.

대표 또는 중간관리자가 바로 봐야 하는 것:

- 현재 회사 건강도
- 매출 흐름
- 거래처 수와 활성도
- 거래처 분포도
- 거래처별 히스토리
- 이탈 위험
- 품목별 변화
- 지역별 확장 기회
- 오늘/이번주 액션
- 지도 기반 거래처 시각화

대시보드 문장 기준:

> 현재 우리 회사가 어떤 상태이고, 다음에 어디를 공략해야 하는가?

## 6. 관리자 화면 기준

관리자는 DB와 업로드 상태를 즉시 확인할 수 있어야 합니다.

필수 표시:

- Supabase 연결 여부
- 테이블별 데이터 건수
- 마지막 업로드 시간
- 마지막 업로드 유형
- 마지막 업로드 파일명
- 마지막 리포트 생성 시간
- 실패한 업로드/오류 메시지
- ERP 매핑 프리셋 수
- 티맵 경로 캐시 수

관리자 화면 문장 기준:

> 지금 데이터가 언제, 어떤 파일로, 어느 테이블에 업데이트됐는가?

## 7. 지금 만들 순서

다음 기능 개발은 아래 순서로 진행합니다.

1. 첫 화면 문구와 업로드 플로우 재정리
2. ERP별 매핑 프리셋 저장/불러오기
3. 업로드 결과 요약: 신규/업데이트/오류/마지막 업데이트 시간
4. 관리자 업로드 이력 상세 화면
5. 티맵 거리/시간 계산 API와 캐시 저장
6. 대표 대시보드 매출/이탈/품목 분석 강화
7. 사업자 상태 정기 체크 설계

## 8. v1에서 미루는 것

아래는 지금 당장 만들지 않습니다.

- 완전한 지도 HeatMap
- 복잡한 권한 체계
- 결제
- 모바일 앱
- 실시간 사업자 자동 조회 배치
- 고도화된 AI 예측 모델

단, DB 구조는 이후 확장을 막지 않도록 준비합니다.
